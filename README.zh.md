# AutoLXB

**实验性安卓端自动化框架，专注于高频、线性的日常任务自动执行**

[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)
[![Android 11+](https://img.shields.io/badge/Android-11%2B-34A853?logo=android&logoColor=white)]()
[![Latest Release](https://img.shields.io/github/v/release/wuwei-crg/AutoLXB?label=Release)](https://github.com/wuwei-crg/AutoLXB/releases)

[English](README.md) | **中文**

</div>

AutoLXB 是一个运行在 Android 真机上的端侧自动化框架。项目采用 **Route-Then-Act** 设计：任务先尝试复用本任务自己的路线资产完成页面跳转，路线无法覆盖的动态交互再交给视觉模型根据截图与界面状态执行。

该项目主要面向重复、线性、可触发的手机任务，例如定时签到、收到通知后回复、进入固定页面读取或提交信息。它不追求让模型无约束地探索整个手机，而是把可复用的页面跳转沉淀为任务路线，把不可预测的页面内容留给视觉执行。

## 软件展示与功能概览

![软件展示](resources/software_ch.png)

核心能力：

- **快速任务**：输入一句自然语言需求，立即执行一次自动化任务。
- **定时任务**：支持单次 / 每天 / 每周执行，可在任务列表页启用或停用。
- **通知触发任务**：根据通知包名、标题、正文和可选 LLM 条件触发任务。
- **任务路线**：每个任务维护独立的 task route，执行时按任务身份命中，不按相似描述复用其它任务路线。
- **路线编辑**：任务执行后可查看捕获到的轨迹，手动删减噪声步骤并保存为该任务的正式路线。
- **任务导入 / 导出**：定时任务和通知触发任务可导出为便携任务 JSON；导入会创建新的本地任务并绑定导入路线。
- **视觉执行**：路线无法完成的步骤由 VLM 观察当前截图、判断下一步动作并执行点击、滑动、输入、返回等操作。
- **Root / Wireless ADB 启动**：Root 设备可直接启动 core；未 Root 设备通过无线 ADB 配对与启动。
- **Trace 日志**：按结构化事件查看 core 运行状态、状态机切换、路线回放、视觉动作和错误原因。

## 工作原理

AutoLXB 的核心运行单元是 `lxb-core`。前端 App 负责任务管理、配置、启动和日志展示；`lxb-core` 作为 `app_process` 后台进程执行任务状态机、页面路由、视觉动作和任务路线读写。

![任务状态机](resources/FSM.png)

任务进入 core 后按状态机执行：

1. **初始化变量**：读取屏幕尺寸、当前 Activity、应用列表、输入能力、解锁策略等运行上下文。
2. **任务路线命中检查**：根据任务身份计算 `route_id`，如果该任务已保存可用路线且路线开关开启，就恢复路线中的子任务结构；如果没有命中，进入普通任务流程。
3. **任务拆分**：把用户任务拆成若干可顺序执行的子任务；拆分失败时退化为单个子任务。
4. **应用解析**：确定当前子任务的目标应用；如果路线命中并且路线片段已经记录了包名，可跳过这一步。
5. **设备准备**：启动或切换到目标应用，必要时完成解锁等准备动作。
6. **页面路由**：优先回放任务路线中的页面跳转步骤；路线完成、路线缺失、路线失败都会产生明确分支。
7. **视觉执行**：基于当前界面截图、任务描述和历史上下文让 VLM 决策动作，直到子任务完成、失败或达到次数上限。

路线命中时，任务会尽量走确定性的路线回放；路线缺失或路线不可恢复时，任务会进入视觉执行。视觉执行期间产生的动作记录也会被用于生成任务路线。

## 任务路线生成过程

![任务路线生成过程](resources/task_map_generation.png)

任务路线由一次真实任务执行沉淀而来。执行过程中，系统同时记录两类信息：

- **视觉侧信息**：截图、模型观察结果、模型给出的动作、动作预期结果。
- **结构侧信息**：当前 XML / accessibility 节点、bounds、text、content description、resource id、class、父级特征等。

当视觉模型输出一次点击、输入或滑动后，core 会把动作结果、截图观察和 XML 结构融合起来。对于点击动作，系统会尽量构建可复用的定位器（locator），并把 locator、容器探测信息、点击坐标和语义说明一起记录到动作轨迹中。保存路线时，这些动作会被整理成任务路线片段和步骤。

任务路线的定位策略按稳定性分层：

1. **locator 定位**：优先使用 text、content description、resource id、class、父级关系、同类索引等 XML 特征定位控件。
2. **容器探测**：locator 不足以定位时，使用点击点附近的可点击容器特征进行校验。
3. **本机坐标兜底**：仍无法稳定表达的控件保留本机点击点或兜底点。
4. **语义适配**：便携导出时，本机坐标类点击会转成语义描述；导入到另一台设备后，首次回放会通过视觉模型把语义描述适配成本机 locator / 容器 / 坐标步骤，并写回本机路线资产。

这种设计让本机重复执行尽量走确定性定位，也让导出的任务在其它设备上有机会通过语义节点完成本机适配。

## 产品形态

App 主要分为四个页面：

- **首页**：启动 / 停止 core，查看运行状态，进入首次配置。
- **任务页**：管理快速任务、定时任务、通知触发任务和最近运行记录；自动化区域旁提供便携任务导入入口。
- **配置页**：配置操控方式、设备端 LLM / VLM、解锁与锁屏策略、语言和可选路线来源。
- **日志页**：查看结构化 trace，打开事件详情，导出本地 trace。

任务页的自动化能力包括：

- 快速任务：一次性执行，不依赖预先保存的任务配置。
- 定时任务：按设定时间或周期触发，任务配置拥有自己的稳定 ID。
- 通知触发任务：按通知规则触发，规则拥有自己的稳定 ID。
- 最近运行记录：查看运行结果和 trace 信息。

## 环境要求

开始前请确认：

- Android **11（API 30）** 及以上真机。
- 使用 **Wireless ADB** 启动时：已打开开发者选项、USB 调试、无线调试。
- 使用 **Root** 启动时：设备已 Root，且能授予 `su` 权限。
- 已配置兼容 **OpenAI Chat Completions** 风格的 LLM / VLM 接口。
  - App 会补齐 `/chat/completions`。
  - 配置页会实时展示最终请求地址。
- 所选模型需要支持图像理解能力，否则视觉执行和语义适配无法正常工作。

## 快速开始

### 1. 安装与手机准备

1. 从 [Releases](https://github.com/wuwei-crg/AutoLXB/releases) 安装 APK。
2. 打开手机开发者选项，并确保以下开关已打开：
   - `USB 调试`
   - **必须开启 USB 调试，否则后台进程可能无法保活**
   - 无 Root 设备还需要打开 `无线调试`
3. 某些国产 ROM 需要额外设置：

   | ROM | 操作 |
   |-----|------|
   | MIUI / HyperOS（小米、POCO） | 开启 `USB 调试（安全设置）` |
   | ColorOS（OPPO / OnePlus） | 关闭“权限监控” |
   | Flyme（魅族） | 关闭“Flyme 支付保护” |

4. 建议将 `AutoLXB` 的电池策略设为 **无限制**，避免后台任务被系统杀掉。

### 2. 启动 AutoLXB Core

- **有 Root**：在首页点击 **Root 启动**，确认系统可以授予 `su` 权限。
- **无 Root**：在首页点击 **ADB 启动**，按引导完成一次无线 ADB 配对。
- 配对完成后，后续通常只需要保持无线调试打开，再点击 **ADB 启动**。

### 3. 配置模型

进入 `配置 -> 设备端 LLM 配置`，填写：

- `API Base URL`
- `API Key`
- `Model`

配置后点击测试，确认模型可以处理图片并返回结果。

### 4. 创建自动任务

AutoLXB 适合“重复、线性、可触发”的任务。建议优先使用下面两类任务：

- **定时任务**：在指定时间或周期自动执行，例如每天早上打开某 App 完成签到。
- **通知触发任务**：监听指定 App 的通知，满足条件后自动执行，例如收到某个群聊消息后自动回复。

任务描述建议写得具体，例如：

```text
打开某 App，进入签到页面，完成签到
打开微信，进入某个群聊，回复刚刚发消息的人
打开外卖 App，进入订单页面，查看骑手位置
```

如果不确定任务描述是否稳定，可以先用 **快速任务** 试跑一次。确认任务路径稳定后，再保存为定时任务或通知触发任务。

### 5. 可选：保存任务路线

对于经常重复执行的任务，可以开启任务路线。任务执行后，在路线编辑页保留有用步骤、删除无关动作，然后点击 **手动保存路线**。后续执行会优先按该任务路线跳转，减少视觉模型调用和不确定性。

## 任务能力

### 快速任务

在首页或任务页输入一句自然语言需求即可执行一次，例如：

```text
打开微信，给文件传输助手发送一条消息：hello
打开 Bilibili，发布一条动态，标题是 test，内容是 test
```

快速任务适合试跑、临时操作和验证模型配置。

### 定时任务

定时任务包含：

- 任务名称
- 用户任务描述
- 可选目标包名
- 可选 Playbook
- 单次 / 每天 / 每周计划
- 是否录屏
- 是否按任务路线执行

定时任务保存后拥有稳定的 `schedule_id`。该任务的路线 ID 为：

```text
schedule:<schedule_id>
```

修改任务描述、执行时间或周期不会改变 `schedule_id`，因此不会因为时间配置变化导致已保存路线失效。导入的定时任务默认处于停用状态，用户确认配置后再启用。

### 通知触发任务

通知触发任务包含：

- 包名匹配（必填）
- 标题匹配 / 正文匹配（可选）
- LLM 条件判断（可选）
- 任务重写策略（可选）
- 命中后的自动化任务描述
- 是否按任务路线执行

通知触发规则保存后拥有稳定的规则 ID。该规则的路线 ID 为：

```text
notify:<rule_id>
```

通知触发链路：

1. core 读取通知快照。
2. 按规则匹配包名、标题、正文和 LLM 条件。
3. 命中后生成自动化任务。
4. 自动化任务进入 `CortexFsmEngine`。
5. 状态机按该规则的 `notify:<rule_id>` 查找并回放任务路线。

## 任务路线（Task Route）

AutoLXB 使用任务级路线资产。路线不按 App 全局共享，也不按任务文本哈希合并；每个任务配置拥有自己的路线 ID 和自己的路线文件。

### 路线身份

路线身份由任务来源决定：

| 任务来源 | 路线 ID |
|---------|---------|
| 定时任务 | `schedule:<schedule_id>` |
| 通知触发任务 | `notify:<rule_id>` |
| 快速 / 手动任务 | 任务运行 ID |

路线文件存储在设备端应用数据目录下的 `lxb_state/task_maps` 中，包含正式 task map、最近一次执行轨迹、最近一次成功轨迹和索引信息。存储文件名会做安全化处理，但路线语义身份仍然是上表中的 `route_id`。

### 路线回放

任务路线开启时，状态机会在任务拆分前查找该任务的 `route_id`：

- 命中可用路线：恢复路线中的子任务与页面跳转步骤，优先执行路线回放。
- 未命中路线：进入普通任务拆分、应用解析、页面路由和视觉执行流程。
- 路线回放失败：记录失败原因，并降级到视觉执行。
- 路线回放成功且路线配置了 **回放后直接结束任务**：任务直接完成，不再进入后续视觉执行。

“回放后直接结束任务”属于路线行为配置，和手动路线一起保存。它控制路线回放成功后是否直接结束任务。

### 路线编辑

路线编辑页用于审查一次任务执行捕获到的轨迹：

- 可查看最近一次捕获轨迹和最近成功轨迹摘要。
- 可删除无关动作。
- 可点击 **手动保存路线** 将保留的动作整理成正式 task route。
- 可删除该任务已保存的路线。

路线编辑页负责编辑路线本身；便携任务导入 / 导出入口放在任务管理和任务配置位置。

## 便携任务导入 / 导出

AutoLXB 的导入导出对象是“任务 + 路线”的便携资产，而不是单独一段坐标脚本。

### 导出

导出入口位于具体任务配置页：

- 定时任务在定时任务配置页导出。
- 通知触发任务在通知触发配置页导出。

导出的 JSON 文件保存到：

```text
Downloads/LXB/Tasks/lxb-task-portable-<时间戳>.json
```

导出文件包含：

- `schema = task_route_asset.v1`
- `task_type`：`schedule` 或 `notify_trigger`
- `task_info`：任务名称、任务描述、包名、Playbook、路线模式等必要信息
- `task_config`：创建新任务所需的干净配置
- `finish_after_replay`
- `segments`：任务路线片段与步骤

导出文件不携带源任务 ID、源 route ID、运行记录、运行时间戳、定时计划时间、通知活跃时段等与原设备或原时间安排绑定的信息。

对于路线中的点击步骤：

- 有 locator 的点击会导出为 `local_locator`。
- 只有容器探测或坐标兜底的点击会导出为 `semantic_tap`，携带目标名称、操作说明、页面上下文和预期结果。
- 导出不把本机坐标当作跨设备事实；坐标类点击需要在目标设备上重新适配。

### 导入

导入入口位于任务页的自动化区域旁。用户选择便携任务 JSON 后，App 会自动识别任务类型：

- `schedule`：创建一个新的定时任务，默认停用，并将路线绑定到新的 `schedule:<schedule_id>`。
- `notify_trigger`：创建一个新的通知触发规则，默认停用，并将路线绑定到新的 `notify:<rule_id>`。

导入后，locator 步骤可以直接参与回放；`semantic_tap` 步骤会在首次回放时调用视觉模型观察目标设备当前界面，生成本机可执行的 locator / 容器 / 坐标步骤，并写回本机路线。适配失败会记录到 trace 和路线状态中。

## 首次配置建议

完成启动后，建议检查以下配置。

### 1. 操控方式配置

这里决定设备如何执行点击、滑动和输入：

- **触控模式**：`Shell` / `UIAutomator`
- **输入模式**：优先推荐安装 **ADB Keyboard**
- **任务期间免打扰**：不处理 / 关闭声音提醒 / 完全静音

### 2. 设备端 LLM 配置

需要填写：

- `API Base URL`
- `API Key`
- `Model`

配置页支持：

- 真实请求 URL 实时预览
- 多份本地 LLM 配置保存与切换
- API Key 密码样式显示
- 模型测试并同步到设备

### 3. 解锁与锁屏策略

- 是否在路线执行前自动解锁。
- 是否在任务结束后自动锁屏。
- 锁屏密码 / PIN（仅在单纯上滑不足以解锁时使用）。

### 4. Playbook

Playbook 是写给自动化执行器的补充操作说明。对于没有稳定路线、页面文案不明显或需要注意特殊流程的 App，可以在任务配置中提供简短明确的 Playbook。

## Trace 与调试

日志页提供结构化 trace 视图：

- 每条 trace 事件单独显示为卡片。
- 点击卡片可查看结构化详情。
- 默认优先加载最新 trace。
- 向上滑动时按需加载更早记录。
- 支持导出本地缓存 trace。

调试 FSM 状态切换、通知触发链路、路线回放、语义适配或视觉执行失败时，trace 中的事件名、`task_id`、`route_id`、状态、失败原因和步骤摘要是主要依据。

## 使用建议

- 将 `AutoLXB` 的电池策略设为 **无限制**。
- 任务描述尽量短、具体、顺序明确。
- 定时任务和通知触发任务适合长期使用；快速任务适合试跑。
- 保存路线前先审查轨迹，删除启动页、弹窗处理等与任务主路径无关的动作。
- 对跨设备导出的任务，优先选择 text / content description / resource id 明确的 App；纯图标、无语义 XML 的控件需要依赖语义适配，稳定性取决于模型识别结果和页面一致性。
- 部分 ROM 更适合 `Shell` 触控，部分更适合 `UIAutomator`，可在配置页切换测试。

## 开发者调试（本地联调）

改完代码后，可直接安装 Debug 版到手机：

1. 连接设备，确认 `adb devices` 能看到设备。
2. 进入 `android/LXB-Ignition`。
3. 执行：

```bash
./gradlew :app:installDebug
```

安装完成后，直接打开手机上的 Debug 版 `AutoLXB` 进行调试。
